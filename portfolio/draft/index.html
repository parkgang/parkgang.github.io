나눔셈 모던화 프로젝트​

기존에 있던 비영리회계 ERP 제품의 퍼포먼스 이슈를 해결하기 위해서 시작되었습니다.
회계 작업을 위한 웹앱으로서 사용자가 회계 작업을 편리하게 할 수 있도록 기능이 제공됩니다.
당시 모던한 기술과 개발 프로세스를 이용하여 이것이 현대적인 개발이구나를 느낄 수 있었습니다.

  • tRPC 도입
    • AS-IS
      • Next.js 에서 HTTP API 를 정의하여 사용하고 있었는데 HTTP API 를 만들고 매번 Client 에 연결하는 것이 불편하다고 느낌
      • 또한, HTTP API 를 변경하면 Client 도 변경해줘야 하고 휴먼 에러 시 엡이 깨지고는 현상이 빈번이 발생
    • Challenge
      • 외부에 사용되는 API 가 아니기에 RPC 를 사용하는 좋겠다고 판단
      • 관련하여 tRPC 를 알게되었고 Server 에서 한번 정의 시 Client 에 손쉽게 연결할 수 있고 게다가 react-query 까지 한번에 연결되어 적합한 기술로 판단
    • TO-BE
      • Next.js 에 정의된 HTTP API 를 모두 걷어내고 tRPC 로 대체함
      • API Spec 변경 시 Client 에 바로 반영되어 개발 생산성 증가
      • 코드 레벨로 호출 경로가 지정되어 자신감있는 리펙토링이 가능하여 유지 보수성 증가
      • Next.js 에서 Server 측 미들웨어 이 부족하여 HTTP Hander 마다 인증 로직을 추가하는 것이 불편했는데 tRPC 의 미들웨어 를 통해 응집성 을 보강
  • 팀으로써 일한다는 것이 무엇인지, 어떻게 협업해야 하는지
    • AS-IS
      • 새로운 기술 도입 시 러닝 커브가 빈번하게 발생
      • 기술을 습득하는 것이 개인의 몫이라고 생각하고 있었음
    • Challenge
      • 새로운 기술을 도입의 시기 조절
      • 팀원들이 손쉽게 습득할 수 있도록 샘플 코드 제작
    • TO-BE
      • 샘플 코드를 통해 팀원이 손쉽게 습득할 수 있도 그에 맞게 코드가 개선됨
      • 기술 리더라면 기술을 더 쉽게 이해할 수 있도록 도와주는 것이 중요하다는 것을 알게됨
      • 비단, 소프트웨어 엔지니어 뿐 아니라 프로젝트를 진행하는 모든 사람들이 그렇다는 것을 알게되었습니다.
  • 왜, 각 레이어를 분리해야 하는지 (e.g. 비즈니스 로직 분리)
    • 이론적으로 비즈니스 로직 을 분리하는 것이 좋다 라는 것을 알고 있었지만 추상적이여서 분리하지 않으면 어떻게 되는지 궁금했습니다.
    • 프로젝트를 진행하면서 비즈니스 로직 이 명확하게 뽑이지 않아 여러 곳에 분리하게 되었습니다.
    • 어디는 Client 에서 어디는 Server 에서 어디는 Database 에서
    • 게다가 Server 처리되어야 하는 것 또한 Client 에서 처리되는 경우도 있었습니다.
    • 이렇게 되면 기대하는 곳에 로직 이 존재하지 않아 찾아다녀야 했고 레이어 분리가 정확히 되지 않아 수정 시 Side Effect 를 예상하지 못하는 경우가 많았습니다.
    • 레이어 분리는 레이어 에 맞게 여러 개발자가 협업할 때 원하는 곳에 로직이 있도록하고 레이어 에서 책임질 수 있도록 단일 책임 원칙 을 고수할 수 있다는 것을 느꼈습니다.
    • 당시 레이어드 아키첵처 와 도메인 주도 설계(Domain Driven Design, DDD) 에 대해서 관심있게 찾아보고 적용을 시도해볼 수 있었습니다.
  • 왜, KISS(Keep it small and simple) 가 중요한지
    • UI 를 그리기 위해서 데이터를 데이터의 명확성을 중점으로 디자인을 하였습니다.
    • API Level 에서는 명확하게 필요한 인터페이스 를 디자인했다고 생각했지만 실제로 UI 를 그리면서 불편하다는 것을 느낄 수 있었습니다.
    • API 소비자에 맞게 디자인되지 않는 것도 문제이었지만 명확한 구조가 아닐지언정 UI 를 그리기 위해서 멍청하게 데이터를 가공하는 것이 좋다는 것을 느낄 수 있었습니다.
    • API 소비자는 한명인데 여러 명이 쓰는 것처럼 복잡하게 설계하고 있다는 것을 알 수 있었습니다.
    • 이외도 프로젝트 스트럭처를 너무 세세하게 분리하니 협업 시 불편함을 느낄 수 있었습니다.
  • Client-Server-Model 에 대한 Insight가 깊어지고 제품 설계 능력에 대해서 큰 시야 경험
    • Client 와 Server 를 같이 개발하면서 벨리데이션 체크 과 로그 는 누가 책임져야 하는가? 고민이 있었습니다.
    • 두 곳 모두에 정의히면 중복 코드라는 느낌이 너무 강하게 들었기에 한 영역에서 모두 책임지는 것이 더 좋다고 생각해왔습니다.
    • 하지만, 각각의 영역은 단일 책임 원칙 을 지켜야 하기 때문에 각각의 영역에서 책임져야 한다는 것을 알게되었습니다.
    • Client 는 사용자 경험 에 초점을 맞춰서 로그 와 메트릭 을 남겨야할 것이고, Server 는 퍼포먼스 가 문제가 없는지 보안 문제는 없는지를 예시로 들 수 있습니다.

현대백화점​

Microsoft 365를 도입하면서 아쉬운 부분을 해결하기 위해 MS Teams Third Party 여러 웹앱을 개발한 프로젝트입니다.

개발된 제품은 아래와 같아요.

  • 팀즈 보안 파일 뷰어 (DRM 및 MIP으로 암호화된 파일을 복호화 해서 파일 미리 보기를 제공)
    • 스켈레톤 UI, ProgressBar를 이용하여 Non-Blocking 되도록 UX 대폭 개선
      • 기존에는 클릭 Action마다 화면 전체에 로더를 표시하여 오랜 시간 동안 사용자가 다른 Action을 할 수 없던 것을 화면 최상단에 ProgressBar를 표시하여 요청이 실행 중임을 나타내고 그동안 Non-Blocking이 되어 다른 Action을 추가적으로 진행할 수 있도록 개선
      • 베타 테스트 결과 실제 API 속도 개선이 이뤄진 것은 아니지만 UX으로 인하여 속도가 매우 빨라졌다는 경험을 전달할 수 있었음
    • 스파게티 React 코드 분석 후 최적화
      • 여러 군데에서 queryString을 조회하는 코드를 단일 책임 원칙을 지키도록 Custom Hook으로 제작
      • 불필요한 Prop Drilling을 Global Status으로 전환하여 중첩 구조 최적화
      • 불변성을 지키지 않는 state code 픽스
  • 팀즈 바로 가기 (외부 시스템에 편하게 접속할 수 있는 바로 가기 링크 제공)
    • 링크 클릭 시 Token이 발급되고 Token을 검증할 수 있는 API를 제공
    • 여러 엔지니어가 거쳐간 제품으로 스트럭처가 일정하지 않아 유지 보수성이 떨어지는 제품이었는데 해당 프로젝트에서 진행된 React 제품들과 유사하게 리펙토링을 통해 후임 엔지니어의 유지 보수성 개선
    • API 인증 KEY와 요청받을 수 있는 IP를 제한하여 보안적으로 더 단단한 API를 설계해야 한다는 것을 알 수 있었음
  • 팀즈 조직도 (로그인한 사용자 권한에 맞게 조직도 정보를 조회/탐색/검색 기능 제공)
    • 하이어라키 구조를 Tree 형태로 표현하도록 UI 제작
    • 정보 클릭 시 Table 구조로 사용자 정보를 보여주도록 제작
    • 검색 시 사용자 정보를 하이라이팅 하여 보여주며 PC에서는 more으로, Mobile에서는 무한 스크롤로 탐색 제공
    • Mobile에서 그룹 채팅 초대 UI를 위해 Drawer UI PoC 후 적용
    • 사용자 사진 동기화를 위하여 ASP.NET Core 백그라운드 작업을 이용하여 동기화 코드 제작
    • 사용자 사진 HTTP 캐시 설계
    • 첫 개발 당시 느슨하게 잡힌 타입을 강하게 잡아서 유지 보수성을 개선

교보생명​

Microsoft 365를 도입하면서 아쉬운 부분을 해결하기 위해 MS Teams Third Party 여러 웹앱을 개발한 프로젝트입니다.

개발된 제품은 아래와 같아요.

  • 팀즈 전자결재 (고객 측 API를 통해 Teams에서 전자결재를 처리할 수 있도록 제공)
    • 외부의 API 직접 소비해 보면서 문제가 없는지 검증하고 소비가 불편한 interface도 경험하면서 어떻게 설계하는 것이 올바른가 생각할 수 있었음
    • Client Side, Server Side 로그를 직접 심어보면서 어떻게 남겨야 추적성이 좋을까? 필요한 정보를 추출할 수 있을까? 로그 설계의 어려움을 경험
  • 팀즈 다운로드 (직접 다운로드를 금지한 오피스 클라우드의 파일에서 MIP를 해제하고 DRM을 걸어 Teams 앱 내부에서 다운로드할 수 있도록 제공)
    • DRM과 같이 보호된 솔루션을 클라우드 환경의 다른 SaaS(M365, Teams)에서 다운로드할 수 있는 설계를 이해하고 구현한 경험
    • 자연스럽게 MIP(Microsoft Information Protection)도 알게 되며 파일 Level, OS Level, 등 여러 종류의 암호화가 있다는 것을 알 수 있었음
    • 이 과정에서 DRM 모듈을 사용해 보며 어떤 암호화 기능이 있는지 알 수 있었음
    • 고객이 DRM을 왜 사용하는지, 해당 제품을 도입해서 얻고 싶은 가치가 무엇인지, 경험할 수 있었음
