프로젝트 상황에 맞게 tRPC 도입​

  • AS-IS
    • Next.js 에서 HTTP API 를 정의하여 사용하는데 만든 것을 Client 에 매번 연결하는 것이 불편하다고 느낌
    • 또한, interface 가 변경되면 그에 맞게 Client 도 대처가 필요했는데 실수로 누락하는 휴먼 에러 발생 확률이 높았음
  • Challenge
    • 외부에 사용되는 API 가 아니기에 RPC 와 같은 기술을 사용하는 것이 좋겠다고 판단
    • 관련하여 tRPC 를 알게되었고 Server 에서 한번 정의 시 Client 에 손쉽게 연결할 수 있고 게다가 react-query 까지 한번에 연결되어 적합한 기술로 판단
  • TO-BE
    • API Spec 변경 시 Client 에 바로 반영되어 개발 생산성 및 DX 증가
    • 코드 레벨 에서 호출 경로가 지정되어 자신감있는 리펙토링이 가능하여 유지 보수성 증가
    • Next.js 에서 Server 측 미들웨어 기능이 아쉬워서 HTTP Handler 마다 인증 로직을 추가하는 것이 불편했는데 tRPC 의 미들웨어 를 통해 응집성 을 보강
  • Reference
    • PR: tRPC 와 iron-session 함께 사용할 수 있도록 예제 추가

안정적인 출시를 위한 husky 정적 도구 도입​

    • 기존에 알고 있던 husky 를 도입하여 필요한 상황에 맞게 Script 를 작성하면 되겠다고 판단
  • TO-BE
    • 코드 리뷰어 로써 안정적인 코드가 Push 되므로 더 나은 코드 개선점 에 대해서 코드 리뷰 를 진행할 수 있게 됨
    • Type error 와 같은 기본적인 문제가 발생하지 않아 노이즈 없는 CI 문제에 대해서 집중할 수 있게 됨

  • Challenge
    • 어떤 것이 문제인지 생각해 보고 시니어에게도 조언을 구함
    • 새로운 기술을 도입의 타이밍을 조절

  • TO-BE
    • 샘플 코드를 통해 팀원이 손쉽게 습득하고 그에 맞게 코드 퀄리티가 개선됨
    • 기술 리더 라면 기술을 더 쉽게 이해할 수 있도록 도와주는 것이 중요하다는 것을 알게됨

    • 팀 은 같은 목표를 가지고 있는 동료라는 것을 인지하고 행동할 수 있어야 한다는 것을 느낌
    • 이 가치를 알고 서로 의견을 나누고 제품을 개발하며 즐겁고 든든한 경험을 함

왜, 각 레이어를 분리해야 하는지 (e.g. 비즈니스 로직 분리)​

  • AS-IS
    • 회계 에서 기초 데이터로 전표 , 회계 기수 , 거래처 , 등이 있는데 이러한 데이터는 일련의 비즈니스 로직 을 통해서 데이터가 CRUD 됨
    • 반복적인 test 를 위해서 손쉽게 데이터 CRUD 가 필요했는데 이를 위해선 비즈니스 로직 을 태워서 가공되어야 하지만 레이어 가 분리되어 있지 않으니 재사용 성이 많이 떨어졌다.
  • Challenge
    • tRPC 의 프로시저 에 비즈니스 로직 를 응집시켜 놓았기에 필요시에 tRPC Server 를 띄어서 사용할 수 있도록 시도
    • 하지만 tRPC 는 여러 의존성이 많아서 깔끔하게 Server 를 띄어서 동작하기 어려웠으며 비즈니스 로직 을 수행하고 싶은 것인데 Server 를 띄우는 것이 이상하다고 판단
    • 프로시저 의 양이 꽤 되어서 당장 새로운 전환은 어렵기에 임시 방편으로 이렇게 사용했지만 비즈니스 로직 추출이 필요하다고 판단
    • 비즈니스 로직 을 추출하기 위해 레이어드 아키텍처 와 DDD 를 적용해보기로 결정
  • TO-BE
    • DDD 는 모호하게 적용되었지만 레이어드 아키텍처 는 application(use case) , domain(entities, repositories, services) , infrastructure(mock, persistence) , presentation 으로 적용하여 코드 추가 시 규칙 없이 작성되던 부분을 개선할 수 있었음
    • 단, 레이어 간의 추상화 레벨이 모호하여 코드 복잡성 과 의존성 이 높아졌음
    • 그래도 이것을 기반으로 Unit Test 를 시도해 볼 수 있었음
    • 핵심 로직을 분리해야지 여러 interface (e.g. API, Console)에서 사용할 수 있다는 것을 경험

왜, KISS(Keep it small and simple) 가 중요한지​

  • AS-IS
    • 명확한 것이 중요하다고 생각하여 프로젝트 구조 와 API 인터페이스 를 명확하게 디자인해왔음
    • 갈수록 프로젝트 구조 는 복잡해지고 UI 도 그리면서 API 인터페이스 가 명확하다고 생각했지만 실제로 UI 를 그리면서 불편함을 느낌
    • 여기서 불편함을 느낀 것 중 하나로, 조건을 말로 표현하면 한 문장 인데 코드로는 여러 조건 이 필요했음
  • Challenge
    • 단순하게 하려고 해도 단순해지지 않아 시니어에게 조언을 구함
  • TO-BE
    • 현실에 맞게 , 목적에 맞게 디자인해야 하는 것을 알게됨
    • 데이터 값 우뮤에 따른 처리를 예시로 보면 렌더링에 사용되는 데이터의 null 여부로 구분했는데 멍청하게 디자인하면 데이터 우뮤에 따른 속성 을 따로 만들 수 있다. 멍청하게 조건 처리를 위해서 boolean 이 있으니 더 명확해지고 관심사 에 집중할 수 있다. 이 생각이 화면에 그리기 위한 속성으로써의 목적에도 부합하다.
    • 실버 불렛 과 같은 데이터 구조 는 없고 목적 에 맞게 KISS 를 지키는 것이 좋다는 것을 느낄 수 있었음

Client-Server-Model 에 대한 Insight​

  • AS-IS
    • 벨리데이션 은 Client 와 Server 모두 검증이 필요하다.
    • 근데, Client 와 Server 에 모두 벨리데이션 을 정의하니 중복 코드 가 발생하기에 한 영역에서 모두 책임지는 것이 더 좋다고 생각이 들었다.
  • Challenge
    • Client 의 벨리데이션 은 제거하고 Server 의 벨리데이션 만 남겨놓음
    • 문제로 유려한 UX 를 위해서 왜 실패했는지 알아야 하는데 실패 이유는 Server 만이 알고 있으니 Client 에서는 실패 이유를 알 수 없음
    • Client 가 알기 위해서는 Server 에서 관련 정보를 응답 해야 함
    • 이러면서 Server 에서 어떤 필드 인지 인지하고 왜 실패했는지 응답 을 해줘야 하는데 실패 Case가 너무 다양하여 대처가 가능한가 생각하게 됨
    • 필드 와 에러 코드를 이용해서 구현에 성공하긴 했지만 이게 과연 유지 보수 에 도움이 되는가? 생각하게 됨
    • Server 가 Client 에 대해서 알아야 하는 책임이 생겼는데 이 때문에 Server 가 Client 의 눈치를 보면서 수정을 하게 됨을 인지
  • TO-BE
    • 잘못된 구성이라는 것을 알게되었다.
    • Server 가 다하려고 하는 것은 제어의 역전 이다. Client 와 Server 의 역할에 맞게 분리되어야 한다.
    • Full Stack 이라고 해서 Client 와 Server 가 하나가 되는게 아니다.
    • 이 이해를 바탕으로 벨리데이션 더 나아가 로그 도 어떤 목적을 가져야 할지 알게 되어 Client 와 Server를 새로운 관점으로 볼 수 있었다.

주요 구현​

  • react-query 의 prefetch 를 통해 props drilling 를 하지 않고 손쉽게 데이터를 가져올 수 있도록 구조 설계
    • getServerSideProps 에서 prefetch 를 통해 미리 필요한 데이터를 확보할 수 있음
    • 미리 확보된 데이터는 QueryClient 를 통해 얻을 수 있음으로 데이터 전달을 위한 불필요한 Props 가 불필요해짐
    • 덕분에 필요한 Props 노출할 수 있었고 데이터 와 컴포넌트 가 느슨하게 연결할 수 있게됨
  • 로그인 유무에 따른 UI 표시 처리 Insight
    • 로그인 유무는 SSR , CSR 두 곳에서 모두 확인이 가능한데 처음에는 SSR 에서 처리를 하도록 접근
    • 하지만, SSR 에서 처리할 만큼 보안 중요도가 높은 UI 가 아니였으며, 프로모션 사이트 답게 빠른 응답성이 중요하다고 판단
    • 빨리 파일은 서빙하고 CSR 으로 처리하는 것이 사용성 에 좋다는 Insight 를 도출하여 사용자가 더 빨리 화면을 볼 수 있도록 CSR 으로 변경
  • 반응형 이 SSR 에 잘 적용되도록
    • 반응형 을 이전에는 hook 을 이용하여 구현했는데 이는 CSR 에서만 동작하기에 SSR 서빙 후 바로 적용되지 않는 문제가 발생
    • 이를 해결하기 위해서 CSS 적으로 반응형 을 처리가 필요했고 MUI 의 sx 를 이용하여 구현
  • 협업을 위해서 Git flow 를 공유하고 Git Hooks 을 이용하여 프로젝트 정책을 확립
    • 덕분에 코드 리뷰 를 편안하게 진행

주요 구현​

  • 스켈레톤 UI, ProgressBar를 이용하여 Non-Blocking 되도록 UX 대폭 개선
    • AS-IS
      • 화면 왼쪽에서 채널(디렉터리) 를 선택하면 오른쪽에 관련 파일 을 표시해주는 UI/UX 입니다.
      • 기존에는 채널 클릭마다 화면 전체에 로딩 을 표시하여 오랜 시간 동안 사용자가 다른 Action을 할 수 없었습니다.
      • 또한, 파일 은 DRM 해제 후 볼 수 있어서 클릭 후 오랜 시간이 걸린 뒤 화면에 보여집니다.
      • 파일 보기 또한, 화면 전체에 로딩 을 표시하여 오랜 시간 동안 사용자가 다른 Action을 할 수 없었습니다.
    • Challenge
      • 채널 클릭 및 파일 보기 모두 사용자를 막는 Blocking 이 있는 UI 이었습니다.
      • 실수로 클릭하더라도, 취소하고 싶더라도 Blocking 이 모두 끝나야 다른 Action을 할 수 있었습니다.
      • 이는 푸어한 UX 이기에 이를 개선하기 위해서 스켈레톤 UI 와 ProgressBar 를 이용하여 Non-Blocking 구현이 필요하다고 생각되었습니다.
      • 구현 시 뒤로가기 혹은 취소 버튼 시 요청한 HTTP Req 에 대해서 취소하고 적절하게 UI 를 변경해야 합니다.
      • axios.CancelToken 을 이용하여 HTTP Req 를 취소하고 UI 를 변경하도록 구현하였습니다.
    • TO-BE
      • 베타 테스트 결과 실제 API 속도 개선 이 이뤄진 것은 아니지만 UX 으로 인하여 속도가 매우 빨라졌다는 경험을 전달할 수 있었습니다.
  • 스파게티 React 코드 분석 후 최적화
    • 여러 군데에서 queryString 을 조회하는 코드를 단일 책임 원칙 을 지키도록 Custom Hook 으로 제작
    • 불필요한 Prop Drilling 을 Global Status 으로 전환하여 중첩 구조 최적화
    • 불변성 을 지키지 않는 state code 픽스
  • 사용자 사진 HTTP 캐시 설계
    • 사용자 사진 동기화를 위하여 ASP.NET Core 백그라운드 작업 을 이용하여 동기화 됩니다.
    • 동기화 가 주기적으로 이뤄지지 않기 떄문에 조회 후 일정 시간은 캐시 가 필요했습니다.
    • HTTP 캐시 에 대해서 모호하게 알고있는 부분이 있어 간단한 HTTP Cache Server를 만들어보자 (with golang) 를 통해 정리하였습니다.
